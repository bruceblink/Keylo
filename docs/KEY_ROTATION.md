# Keylo 1.1.0 密钥轮换建议

Keylo 1.1.0 已支持通过 `JWT_KEY_ID` 和 JWKS 暴露当前验签公钥，但当前版本的密钥轮换策略仍建议以人工控制的方式执行。

## 轮换目标

密钥轮换的目标是：

- 替换旧私钥，降低长期暴露风险
- 让下游系统可以识别新的 `kid`
- 避免直接共享私钥给任何第三方系统

## 1.0 推荐轮换方式

### 方案一：维护窗口内整体切换

适用于系统数量不多的阶段。

步骤：

1. 生成新的 RSA 私钥和公钥。
2. 将 `JWT_KEY_ID` 更新为新值。
3. 替换 `JWT_PRIVATE_KEY_PATH` 和 `JWT_PUBLIC_KEY_PATH`。
4. 重启 Keylo。
5. 通知下游系统刷新 JWKS 缓存。

这种方式简单直接，但要求下游系统具备较短的 JWKS 缓存周期。

### 方案二：先切流量较低环境验证，再发布生产

推荐顺序：

1. 开发环境验证新密钥
2. 预发环境验证 JWKS 与内省
3. 生产环境低峰切换

## 轮换时的注意事项

1. 新 `kid` 必须唯一。
2. 私钥和公钥必须一一对应。
3. 下游系统应优先通过 JWKS 拉取公钥，而不是手工固定公钥内容。
4. 对高敏接口，轮换期间可优先依赖内省接口降低缓存不一致影响。

## 1.0 的已知限制

Keylo 1.0 当前只对外发布一把活动公钥。这意味着：

- 不支持新旧两把公钥长期并行发布
- 不支持无感平滑双 key 过渡

这不会阻塞 1.0 发布，但属于后续 1.x 版本优先级较高的增强项。

## 对接系统的建议

第三方系统应采用以下策略：

1. 通过 `/.well-known/jwks.json` 获取公钥。
2. 缓存 JWKS，但缓存时间不要过长。
3. 在验签失败且 `kid` 不匹配时，主动刷新 JWKS。
4. 对需要强实时性的接口，回退到内省接口补充校验。
