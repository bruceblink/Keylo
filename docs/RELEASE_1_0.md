# Keylo 1.0 发布说明

Keylo 1.0 是首个可作为统一认证中心正式集成到多系统架构中的版本。

## 版本定位

Keylo 1.0 的目标是提供一套可以被后台管理系统、内部服务和 API 网关稳定接入的认证与鉴权能力，而不是单个系统内嵌的登录模块。

## 1.0 已具备的能力

### 认证能力

- 用户名密码登录
- OAuth 第三方登录
- Access Token / Refresh Token
- 登出与黑名单
- 用户信息查询

### 服务鉴权能力

- 服务注册与服务密钥轮换
- `service_access` Token 签发
- 服务 Token 内省

### 第三方集成能力

- 用户 Token 内省
- RS256 JWT 签发
- `/.well-known/jwks.json` 公钥发布
- 第三方系统接入文档

### 基础运维能力

- PostgreSQL 自动迁移
- 审计日志清理
- 限流与登录锁定
- Docker 开发环境支持

## 1.0 的安全边界

Keylo 1.0 明确不再推荐共享 JWT 签名密钥模式，而是以以下方式作为标准接入路径：

1. 下游系统通过 JWKS 获取公钥并做本地验签。
2. 需要实时吊销判断时，调用内省接口补充校验。
3. JWT 私钥只保留在 Keylo 侧。

## 1.0 的推荐接入方式

### 对后台系统

例如 AgileBoot：

- Keylo 负责认证中心能力
- AgileBoot 负责页面、菜单、角色、数据权限
- AgileBoot 使用 Keylo token 建立身份，再做本地授权

### 对内部服务

- 使用 `/v1/service/token` 获取 `service_access` Token
- 使用 `/v1/auth/introspect` 或 `/.well-known/jwks.json` 验证用户 Token

## 1.0 未覆盖的增强项

以下内容不阻塞 1.0 发布，但属于 1.x 后续增强：

- 多把 RSA 密钥并行发布
- 自动密钥轮换流程
- 更细粒度的健康检查与 readiness 探针
- 更完善的网关接入样例

## 推荐的 1.0 验收结论

当以下条件成立时，可将当前版本视为正式 1.0：

1. RS256 和 JWKS 已启用。
2. 生产环境已显式提供 RSA 密钥与管理员客户端。
3. 用户 Token 和服务 Token 内省均可用。
4. 第三方系统已完成至少一次真实接入验证。
5. 集成测试通过。
